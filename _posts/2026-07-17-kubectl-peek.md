Working with Kubernetes Secrets is easy until you need to answer questions such as:

- **What does this Secret contain?**
- **Which workloads use it?**
- **Was it generated by an operator?**
- **Is it safe to delete?**

Answering those questions usually means switching between several `kubectl` commands, decoding Base64 values manually, grepping manifests, and inspecting CRDs one by one.

I wanted something much simpler.

That is why I built **kubectl-peek**.

`kubectl-peek` is a lightweight interactive CLI and native `kubectl` plugin that helps you inspect Secrets and understand their relationships across your Kubernetes namespace—all from your terminal.

No controllers.

No CRDs.

No web UI.

No installation inside the cluster.

Everything runs client-side using the kubeconfig you already have.

---

# Why I built it

My goal was to build something that stays intentionally small while still being genuinely useful during day-to-day platform engineering work.

I wanted a tool that:

- works with any Kubernetes cluster
- requires no cluster-side installation
- respects your existing kubeconfig and RBAC
- is fast enough to become part of your normal workflow

Instead of copying Secret names into `kubectl get`, decoding values, opening Deployments, checking Pods and hunting through manifests, I wanted everything in one place.

---

# What kubectl-peek does

The tool provides an interactive Secret browser directly in your terminal.

You can:

- browse Secrets
- filter interactively
- inspect decoded Secret values
- discover where a Secret is used
- detect resources that generate Secrets
- extend discovery to your own CRDs
- use it as either `kubectl-peek` or `kubectl peek`

# Built-in Secret discovery

Out of the box, `kubectl-peek` automatically discovers references from common Kubernetes resources.

Currently supported:

- Pods
- Deployments
- StatefulSets
- DaemonSets
- Jobs
- CronJobs
- ServiceAccounts
- Ingresses
- Gateway API Gateways

It detects references through:

- Secret volumes
- projected Secret volumes
- imagePullSecrets
- environment variables
- envFrom
- init containers
- ephemeral containers
- Gateway listener certificates
- Ingress TLS configuration

Example:

```text
Secret: database-credentials

Used by:

Deployment/backend
  uses: container environment (container/backend envFrom)

CronJob/backup
  uses: container environment variable (container/backup env/BACKUP_PASSWORD -> password)

Ingress/web
  uses: TLS certificate (spec.tls[0].secretName)
```

This makes it much easier to understand whether a Secret is still referenced before rotating or deleting it.

# Discover relationships in your own CRDs

The newest feature is support for **custom Secret discovery**.

Instead of hardcoding support for every Kubernetes operator, `kubectl-peek` can load discovery rules from a simple YAML file.

That means you can immediately support resources such as:

- External Secrets Operator
- cert-manager
- Crossplane
- internal platform APIs
- any custom CRD

without changing the application itself.

Example:

```yaml
rules:
  - apiVersions:
      - external-secrets.io/v1
      - external-secrets.io/v1beta1

    kind: ExternalSecret
    resource: externalsecrets

    references:
      - path: spec.target.name
        relation: produces
        description: generated Secret
```

Rules can be supplied on demand:

```bash
kubectl-peek --rules rules.yaml
```

or configured once:

```bash
export KUBECTL_PEEK_RULE_FILE="$HOME/.config/kubectl-peek/rules.yaml"
```

Built-in discovery and rule-based discovery are automatically combined.

# Basic usage

Browse Secrets:

```bash
kubectl-peek
```

Filter by name:

```bash
kubectl-peek database
```

Use another namespace:

```bash
kubectl-peek -n staging
```

Specify a kubeconfig:

```bash
kubectl-peek --kubeconfig ~/.kube/config
```

Load custom rules:

```bash
kubectl-peek --rules rules.yaml
```

---

# Security

`kubectl-peek` displays decoded Secret values directly in your terminal.

Remember that those values may remain visible in:

- terminal scrollback
- recordings
- screenshots
- shared sessions

Use it only with appropriate RBAC permissions and in trusted environments.
